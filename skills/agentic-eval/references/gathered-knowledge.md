# Gathered Knowledge: Agentic Evaluation & LLM-as-Judge

Research synthesized from academic papers (arXiv, ACL, NeurIPS, ICLR 2024–2025) and industry sources.

---

## 1. LLM-as-Judge: Core Biases (Empirically Validated)

### Positional Bias
- LLM judges systematically favor responses that appear in a **specific position** (often first) in pairwise comparisons, regardless of quality.
- Bias magnitude varies by judge model, task type, and candidate quality.
- **Mitigation**: Run pairwise comparisons twice with swapped order, then average. If results conflict, discard and use a different judge or use absolute scoring instead.
- Source: "Judging the Judges: A Systematic Study of Position Bias in LLM-as-a-Judge" (arXiv:2406.07791, IJCNLP 2025)

### Self-Enhancement Bias (Self-Preference Bias)
- LLMs give preferential ratings to outputs generated by themselves or same-family models, even when the outputs are not better.
- GPT-4 prefers outputs with lower perplexity (resembling its own style), not necessarily because they're better.
- GPT-4o and Claude 3.5 Sonnet both show statistically significant self- and family-preference bias.
- **Mitigation**: Never use the same model as both generator and judge. Use a judge from a different model family.
- Source: "Self-Preference Bias in LLM-as-a-Judge" (arXiv:2410.21819, NeurIPS 2024); "Play Favorites" (2025)

### CALM Framework: 12 Identified Biases
The CALM framework (ICLR 2025, "Justice or Prejudice?") identifies:
1. Positional bias
2. Self-preference / self-enhancement
3. Verbosity bias (longer responses rated higher regardless of quality)
4. Alignment bias (safety-tuned models rate safe responses higher)
5. Familiarity bias (prefers common phrasing)
6. Authority bias (defers to human-like framing)
7. Sycophancy (agrees with the requester's implicit preference)
8. Format bias (prefers markdown, bullet points)
9. Language bias
10. Order consistency bias
11. Bandwagon effect
12. Scoring drift across context

**Practical implication**: LLM judges are reliable for *relative ranking* on clear criteria, but unreliable for *absolute scores* or tasks requiring cultural, domain-specific, or safety judgments.

### Verbosity Bias
- Judges consistently rate longer responses higher, even when brevity would be better.
- **Mitigation**: Explicitly instruct the judge to score conciseness as a separate criterion, or set a word-count penalty.

## 2. Rubric Design: Academic Best Practices

### LLM-Rubric (Microsoft Research / ACL 2024)
- **Key finding**: Decompose rubrics into independent single-question prompts, each evaluated separately.
- Calibration via judge-specific neural networks significantly reduces prediction error vs. naive averaging.
- Human-human inter-rater agreement is the practical ceiling for LLM-human agreement — do not expect superhuman rubric scoring.
- Source: "LLM-Rubric: A Multidimensional, Calibrated Approach" (arXiv:2501.00274, ACL 2024)

### Autorubric (arXiv 2025)
- Unified framework supporting binary, ordinal, and nominal rating types.
- Integrates ensemble judging, few-shot calibration, option shuffling, and length penalties.
- Reports Cohen's kappa and Krippendorff's alpha for rubric reliability.
- Source: "Autorubric: A Unified Framework for Rubric-Based LLM Evaluation" (arXiv:2603.00077)

### Key Rubric Design Rules (Evidence-Based)
1. **Prompt each criterion separately** — combining multiple criteria in one prompt reduces reliability.
2. **Calibrate with few-shot examples** — 2–4 examples per criterion dramatically improves consistency.
3. **Report reliability metrics** — always check Cohen's kappa; below 0.6 means the rubric is too ambiguous.
4. **Avoid >5 criteria** — beyond 5, inter-rater reliability degrades and criteria start overlapping.
5. **Binary or 3-point scales outperform 1–10 scales** — fine-grained numeric scales have poor reliability. Use: FAIL/PASS or FAIL/PARTIAL/PASS.

## 3. Agentic Evaluation Patterns: Research Findings

### Reflect-and-Refine (Self-Critique Loop)
- **When it helps**: Tasks with subjective quality where external ground truth is unavailable.
- **Limitation**: Self-critique is systematically softer than external critique. Models tend to approve their own outputs at a higher rate than warranted.
- **Best practice**: Use a *separate* LLM call for critique, not the same context window.
- Source: AWS Prescriptive Guidance on Evaluator/Reflect-Refine Loops; arXiv:2412.04093

### Evaluator-Optimizer (Separate Generator and Evaluator)
- **When it helps**: High-quality bar tasks with clearly defined, actionable criteria.
- **Key requirement**: The evaluator must produce *actionable* feedback — "the accuracy is poor because the dates are wrong" is actionable; "the accuracy is poor" is not.
- **Best practice from Anthropic cookbooks**: Generator and evaluator should be the same model class but prompted differently. Avoid using a weaker model as the evaluator.
- Source: claude-cookbooks/patterns/agents/evaluator_optimizer.ipynb (Anthropic)

### Code Reflection (Test-Driven Refinement)
- **Most reliable pattern** because ground truth (tests pass/fail) is objective.
- Iteration stops definitively when all tests pass — no subjective threshold needed.
- **Critical**: Generate tests *before* writing code (spec-first), not after, to prevent the agent from writing code that passes its own tests trivially.

### Multi-Dimensional Agentic Evaluation (Research Recommendation)
Evaluations of agentic systems should span three layers:
1. **Node-level**: Step utility, tool correctness, individual action quality
2. **Session-level**: Goal attainment, trajectory quality, overall task success
3. **System-level**: Latency, token cost, resource use, reliability

Most single-loop eval patterns only cover node-level. For production agents, add session-level success metrics.

Source: "Survey on Evaluation of LLM-based Agents" (arXiv:2503.16416); "Evaluation and Benchmarking of LLM Agents: A Survey" (arXiv:2507.21504)

## 4. Practical Convergence and Iteration Guidelines

### Choosing max_iterations
| Quality Bar | Recommended max_iterations | Rationale |
|---|---|---|
| Best-effort (subjective polish) | 2–3 | Diminishing returns beyond 3 for subjective tasks |
| Correct output required | 3–5 | Gives meaningful improvement chance while capping cost |
| Must not fail (safety/compliance) | 5–7 + human checkpoint | High stakes justify cost; human review at gate |
| Code with tests | Until tests pass, capped at 5 | Objective criterion — no need for fixed count |

### Convergence Detection
- Stop if score delta < 5% between consecutive iterations (plateau).
- Stop if the *refinement* produces an output more than 80% similar to the previous (not changing meaningfully).
- Always return the best-scoring iteration's output, not the final iteration.

### Score Thresholds
- **0.8+**: "Good enough" for most production use cases.
- **0.9+**: High-quality bar (use for customer-facing content).
- **1.0 / all PASS**: Only for objective criteria like test suites.
- Never use 0.7 as a threshold for anything requiring reliability — too close to random for a 5-point scale.

## 6. Key Gotchas Not Covered by Basic Tutorials

1. **Evaluating evaluation**: Run your evaluator against human judgments on a sample. If Spearman correlation < 0.6, the evaluator is not reliable enough to use autonomously.
2. **Semantic drift in iteration**: After 3+ iterations, outputs tend to get longer and more verbose (verbosity creep), even if not better. Monitor output length across iterations.
3. **Task leakage**: If the generator and evaluator share context, the generator may start optimizing for the evaluator's known criteria rather than the task. Keep evaluation context separate.
4. **Binary is often better than continuous**: A binary PASS/FAIL evaluator with a clear threshold often outperforms a 0–1 score in reliability and actionability. Reserve continuous scoring for comparative ranking only.
5. **LLM judges do not agree with each other**: Across GPT-4, Claude, and Gemini, judge agreement on the same outputs is often only 60–70% on subjective criteria. Use consensus (majority vote) for high-stakes evaluations.

## 7. Additional Critical Gotchas (from codebase synthesis)

### Vague Rubric Anchors
"Good = 3, Excellent = 5" is not actionable. Judges interpret differently per run — causing 5–20% score variance on the same output. Use quantitative anchors: "5 = all requirements met with no errors; 4 = missing < 10%; 3 = missing 10–25%."

### Confidence Paradox
Fluent, confident-sounding outputs score higher, even when wrong. "I am certain X" (incorrect) often beats "I think it might be X" (correct) because judges use fluency as a proxy for accuracy. Mitigation: add an explicit "claims certainty only when justified" criterion.

### Evaluation Noise
Same output run through the same evaluator twice can produce 5–20% score variance. Never use single-run scores as a definitive pass/fail. Run 3+ times and average. Weight decisions on the average, not a single score.

### Overfitting to Evaluation Set
If you optimize against the same fixed examples repeatedly, the generator learns to game those specific examples. Use a 60/40 train/validation split: optimize against the train set, select best output by validation score.

### Judge Model Matters
Claude Opus produces significantly more reliable and nuanced evaluations than Haiku for subjective criteria. Haiku is acceptable for quick binary filtering only. Use Sonnet for most production eval loops; Opus for high-stakes subjective assessment.

### Evaluation Is Not the Same as Improvement
A high score does not guarantee the loop improved the output. Log score delta per iteration: if iterations 2 and 3 show identical scores, the `refine()` prompt is not producing different outputs — the loop is running without doing anything useful.

### Infinite Context Consumption
Evaluation loops at scale are expensive: 5 iterations × 3 rubric calls × 3K tokens = 45K tokens per task. Mitigate: use convergence detection aggressively; batch evaluations where possible; prefer structured JSON feedback over prose (shorter).

## 8. Industry Best Practices (Anthropic Recommendations)

1. **Blind comparison > unblinded**: Never tell the judge which approach generated which output. Show only "Option 1 / Option 2."
2. **Multi-dimensional rubrics > single assertion**: `assert "name" in output` misses all other quality dimensions.
3. **Always include a baseline**: "Scored 0.7" is meaningless without context. "Scored 0.7 vs baseline 0.5 (+40%)" is actionable.
4. **Log everything**: task, iteration, output, rubric, evaluation result, feedback, model, temperature. Debugging eval loops without logs is impossible.
5. **Validate against human judgment**: Spot-check 10% of outputs with humans. If Pearson correlation between eval scores and human ratings < 0.7, the rubric is misaligned — fix it before scaling.
6. **Explicit negative criteria**: Rubrics should penalize bad behavior, not just reward good behavior. Always include at least one "contains no X" criterion.
